# Fix README Reward Explanation Inconsistency

## 引言

在项目文档的“1 CPU = 1 Vote”部分，我们注意到一个影响新用户理解的逻辑不一致问题。该部分首先声明奖励在矿工之间平均分配，然后乘以一个“存续时间乘数”（antiquity multiplier）。然而，后续的纪元奖励示例却显示了每台机器不同的奖励金额，且未提供清晰的公式来连接这两个描述。这可能导致新用户困惑：奖励到底是平均分配还是按机器差异计算？为了解决这个问题，本文档将添加一个简单的内联公式，使推导过程清晰透明。

## 问题分析

### 当前描述与示例的冲突

在“1 CPU = 1 Vote”部分，原文描述大致如下：
- 奖励在所有矿工之间平均分配。
- 然后，每个矿工的奖励乘以一个基于机器存续时间的乘数。

但在纪元奖励示例中，我们看到了类似这样的数据（为说明问题，使用假设数值）：

| 机器ID | 存续时间（天） | 存续时间乘数 | 奖励金额（代币） |
|--------|----------------|--------------|------------------|
| A      | 30             | 1.0          | 10.0             |
| B      | 60             | 1.5          | 15.0             |
| C      | 90             | 2.0          | 20.0             |

这里，机器A、B、C的奖励金额各不相同，与“平均分配”的描述直接矛盾。实际上，奖励是“按比例分配”的，其中比例由存续时间乘数决定，而不是先平均再乘以乘数。

### 根本原因

问题在于“平均分配”一词的误用。正确的机制是：**总奖励池按每个矿工的存续时间乘数权重进行分配**，而不是先平均再缩放。乘数的作用是调整每个矿工在总奖励池中的份额，而不是在平均分配后额外缩放。

## 解决方案：添加内联公式

为了消除歧义，我们将在“1 CPU = 1 Vote”部分添加一个清晰的内联公式，明确奖励的计算方式。公式如下：

### 奖励计算公式

设：
- `R_total` = 纪元总奖励（代币）
- `M_i` = 矿工i的存续时间乘数（基于机器存续时间）
- `n` = 参与该纪元的矿工总数

则矿工i获得的奖励为：

```
reward_i = R_total * (M_i / Σ(M_j for j=1 to n))
```

其中 `Σ(M_j for j=1 to n)` 是所有矿工存续时间乘数之和。

### 公式解释

- 分母 `Σ(M_j)` 是所有矿工乘数的总和，代表总权重。
- 分子 `M_i` 是矿工i的权重。
- 奖励与矿工的权重成正比，而不是平均分配。

### 与示例的对应

使用上表的示例数据，假设 `R_total = 100` 代币：
- 总乘数和 = 1.0 + 1.5 + 2.0 = 4.5
- 机器A奖励 = 100 * (1.0 / 4.5) ≈ 22.22 代币
- 机器B奖励 = 100 * (1.5 / 4.5) ≈ 33.33 代币
- 机器C奖励 = 100 * (2.0 / 4.5) ≈ 44.44 代币

注意：示例中的10.0、15.0、20.0仅用于说明比例关系，实际计算需使用完整公式。

## 文档更新建议

### 修改位置

在“1 CPU = 1 Vote”部分，将原“平均分配”的描述替换为以下内容：

> 每个纪元的总奖励按矿工的存续时间乘数比例分配。具体公式为：
> 
> ```
> reward_i = R_total * (M_i / Σ(M_j))
> ```
> 
> 其中 `M_i` 是矿工i的存续时间乘数，`Σ(M_j)` 是所有矿工乘数之和。这意味着存续时间越长的机器获得越高的奖励份额，但所有奖励的总和始终等于 `R_total`。

### 示例更新

在纪元奖励示例表格下方，添加一行说明：

> 注意：上表中的奖励金额基于公式 `reward_i = R_total * (M_i / Σ(M_j))` 计算。例如，若总奖励为100代币，则机器A的实际奖励为 100 * (1.0/4.5) ≈ 22.22 代币，而非示例中的10.0。示例中的数值仅用于展示比例关系。

## 总结

通过添加内联公式并修正“平均分配”的表述，我们可以彻底消除README中的不一致性。新公式清晰地展示了奖励如何按存续时间乘数比例分配，使新用户能够直观理解计算逻辑。建议立即更新文档，并在用户指南中保留此公式作为参考。
